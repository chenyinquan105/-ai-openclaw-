# 多日旅行智能排程系统 — 架构摘要

## 核心文件
- `skills/multi_day_scheduler/multi_day_scheduler.py` — 排程引擎（~1800行）
- `skills/multi_day_scheduler/scheduling_penalty.py` — 软约束惩罚函数
- `server.py` L482-574 — 排程调度入口；L665-870 — LLM审查；L6904-7208 — /api/smart_schedule 端点

## 五阶段流水线

1. **地理聚类** (KMeans++) — 按经纬度把候选店铺分成N组（N=天数），scipy kmeans2，纯Python兜底
2. **负载均衡** (贪心搬运) — 调平每天时长和景点数，硬规则：每天scenic≤2，饭店/购物不限
3. **每日TSP** (最近邻+2-opt) — 给每天排最优游览顺序，起点终点都是酒店
4. **时间线构建** (~300行核心) — 把顺序变成HH:MM精确时间线，插入午餐晚餐
5. **精修优化** (模拟退火80轮) — 随机微调用餐时间/顺序，最小化代价函数

## 阶段4 关键设计（时间线构建 _build_timeline ~L571-904）

### 品类分类
- 正餐(restaurant/hotpot等) → 不参与VISIT循环，通过"绑定到最近景点"插入午/晚餐
- 非购物(scenic等) → 排在visitable_shops前面，优先占白天
- 购物(shopping) → 排在后面，可晚间

### 吃饭插入逻辑
- 午餐：景点之前插入，用 meal_time_penalty() 选惩罚最小的候选时间（11:00-13:30范围）
- 晚餐：**延迟到景点之后插入**，防止晚餐把景点挤到晚上（之前修过的bug）
- 无餐厅时：插入 LUNCH_NEEDED/DINNER_NEEDED 占位节点

### 时间约束（仅warning，不kill）
- 非购物类在午餐窗口(11:30-13:30)开始 → 推到13:30
- 非购物类排到晚餐后(17:30+) → 记录warning，仍排入
- 营业时间冲突 → 记录warning，仍排入（已关门）或推迟到开门（未开门）
- **核心原则：100%排入，零丢弃**

### 交通速度
- 步行83m/min, 驾车667m/min, 地铁500m/min
- **>3km自动切驾车**（之前bug：颐和园→八达岭45km用步行算9小时）
- 雨天减速(walking_penalty)

## 软约束惩罚体系 (scheduling_penalty.py)

```
总代价 = 跳过惩罚(未访问×200)
       + 用餐时间偏离(舒适区0分, 勉强区线性, 不可接受区加速)
       + 通勤成本(每分钟×0.3)
       + 体力消耗(品类系数×时长×0.5)
```

- 午餐锚点12:00，舒适区11-13点，勉强区9:30-14:30
- 晚餐锚点18:30，舒适区17:30-19:30，勉强区16-21点
- 体力系数：scenic 1.5, shopping 1.0, 餐厅 0.3, cinema 0.5

## 服务端防护 (server.py)

### 洋葱式防护体系
1. 事前：超载检查（scenic > 天数×2 → 拦下来让用户选）
2. 排程中：跨天餐厅去重（同店不在两天出现）
3. 排程后L1：LLM审查（6维检查，auto_fixes + questions + risk_flags）
4. 排程后L2：天气校验（纠正LLM建议"改到下雨天"的bug）
5. 排程后L3：完整性验证（强制检查所有景点都在timeline，缺了就补）
6. 交互时：禁止删除VISIT节点（remove_node对VISIT拒绝执行）

### LLM审查可操作类型
- auto_fixes: move_to_day / swap / reorder / general（禁止remove_poi）
- questions: 选择题让用户决定
- risk_flags: 标记风险不做操作

## 已知问题/优化方向

1. **scenic上限2太死** — 每天硬限制2个景点，没考虑景点大小（逛长城3h vs 逛公园1h）
2. **早餐永远不排** — 全部留占位符，应该能从候选池匹配
3. **聚类不稳定** — KMeans++在数据量小时结果波动大，有时把长城和颐和园分两天
4. **精修层太保守** — 模拟退火只微调时间不跨天移动，跨天全依赖LLM审查
5. **次日HH:MM格式** — _safe_time_str超过24h输出"次日02:11"，后续解析会报错
6. **步行优先是默认** — 大部分用户不改交通方式，导致45km跨城用步行算
7. **LLM审查调用失败时无声** — 空JSON回退，用户无感知
