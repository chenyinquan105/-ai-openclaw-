# 多日行程 — 需求定义 & 总体设计

## 一、这个功能要解决什么问题？

现在美团AI只能规划**一天内**的行程。用户说"明天去理发喝咖啡"，系统排出一条 timeline，从出发到结束全在一天里。

但真实生活中，很多人需要的是**多日规划**，尤其是旅游场景：

**典型场景：**
> 小王周末想去北京玩两天。周六想去故宫、吃全聚德烤鸭、住如家酒店。周日想去颐和园、吃涮羊肉、下午回家。
>
> 他想要的不是两个独立的一天，而是：把所有想去的地方告诉AI，AI自动帮他把店铺分到最合理的日期、每天排最优路线、算好时间——他不用自己琢磨"故宫和颐和园放同一天会不会太赶""全聚德离故宫近还是离颐和园近"。

**核心价值一句话：用户扔一堆想去的店，系统自动算出最省时间的多日安排。**

---

## 二、当前系统的局限（问题定义）

### 问题1：没有"天"的概念

`session_state` 里只有一组 `selected_pairs` 和 `task_list`。所有店铺排进同一条 timeline。如果你选了 8 家店，系统会试图把它们全塞进一天——哪怕时间根本不够。

### 问题2：品类不够，没有旅游相关的东西

当前的品类只有：理发、咖啡、宠物、健身、火锅、日料、电影、干洗、餐饮。

旅游需要的：**酒店（住哪）、景点（玩啥）、早餐、购物**——全都没有。

### 问题3：排程引擎只管排序，不管分天

`_run_schedule_from_session` 的逻辑是：所有 task → 排序 → timeline。它不会思考"这几个放周六、那几个放周日更合理"。

### 问题4：用户得自己操心怎么分天

现在如果你有 8 个想去的地方，你得自己决定哪些周六去、哪些周日去——系统帮不了你。分错了就浪费时间绕路。

## 三、用户故事（怎么用）

### 故事A：游客规划周末两日游

```
1. 打开App，看到顶部有个"📅 单日"按钮，点一下弹出"计划玩几天？"
2. 选了 2 天
3. 开始搜索想去的地方（跟现在一样用）：
   "故宫" → 选了
   "全聚德" → 选了
   "如家酒店三里屯" → 选了
   "颐和园" → 选了
   "涮羊肉" → 选了
4. 底部显示："已选 5 家 | 🪄 智能排程"
5. 点"智能排程"
6. 系统算了几秒，出来结果：

   ┌─ 第1天 ────────────────┐  ┌─ 第2天 ────────────────┐
   │ 09:00 从如家出发       │  │ 09:00 从如家出发       │
   │ 09:15 🏛️ 故宫 (4h)    │  │ 09:30 🏛️ 颐和园 (3h)  │
   │ 13:00 🍽️ 全聚德 (1h)  │  │ 12:30 🍽️ 涮羊肉 (1h)  │
   │ 14:30 自由时间         │  │ 14:00 返程回家         │
   │ 18:00 🏨 回如家        │  │                        │
   │ ⏱️ 约8小时             │  │ ⏱️ 约5小时             │
   └────────────────────────┘  └────────────────────────┘

7. 小王看了看，说："把全聚德换成便宜坊"
   系统把第1天的全聚德换成便宜坊，重排第1天，第2天不变

8. 满意，点"开始行程"，从第1天开始执行
```

### 故事B：单日规划到一半发现需要多日

```
1. 用户已经选了 6 家店，发现一天跑不完
2. 点"📅 单日"→ 弹出"切换到多日？这些店铺将放入候选池"
3. 选 2 天 → 点智能排程 → 系统自动分成两天
```

### 故事C：行中临时调整

```
1. 正在执行第1天行程
2. 突然下暴雨
3. 用户说："把室外景点都挪到明天"
4. 系统把第1天的故宫移到第2天，两天都重新排程
```

## 四、核心设计思路

### 4.1 数据怎么存

现在 session_state 是一个平的结构（只有一组 selected_pairs）。多日需要变成：

```
session_state:
  trip_mode: "single" 或 "multi"   ← 当前是单日还是多日
  trip_days: 2                      ← 计划玩几天
  days: [                           ← 每天的数据
    {
      第1天: selected_pairs + task_list + 排程结果 + 聊天记录
    },
    {
      第2天: selected_pairs + task_list + 排程结果 + 聊天记录
    }
  ]
  active_day_index: 0              ← 当前在看/编辑第几天
```

**关键设计决策：** 单日模式下，所有逻辑跟现在一模一样（向后兼容）。`_apairs()` 辅助函数在单日模式下返回顶层的 selected_pairs，多日模式下返回当前天的——上层代码不需要改。

### 4.2 智能分天引擎怎么工作

这个是整个功能的核心。输入一堆店铺+天数，输出最优分配：

**第1步：地理聚类**
- 把所有店铺的经纬度拿出来
- 按距离聚类（店铺密集的区域自动成一组）
- 比如：故宫、景山、全聚德都在市中心 → 一组；颐和园、圆明园在西北 → 另一组
- 组数 = 天数（用户指定的）

**第2步：每天内部排最优路线**
- 复用现有的排程逻辑
- 起点 = 当天的住宿位置
- 终点 = 当天的住宿位置
- 中间按最近邻贪心排顺序

**第3步：插入吃饭时间**
- 识别哪些店是餐厅（品类=restaurant/hotpot/japanese/breakfast）
- 在时间轴上找合适的午饭窗口（11:30-13:30）和晚饭窗口（17:30-19:30）
- 把最近的餐厅放在这些时间点

**第4步：计算时间**
- 移动时间 = 两点距离 ÷ 交通速度（步行5km/h、地铁30km/h、驾车40km/h）
- 活动时间 = 品类固定时长（景点3h、吃饭1h、购物1.5h）
- 算出来每段啥时候开始、啥时候结束

**第5步：全局微调**
- 看看有没有店铺在两个天的边界上，移到另一边能减少总移动距离
- 交换两天的店铺，算总耗时，用更优的组合

### 4.3 前端怎么展示

**切换按钮：** header 里天气和⚙️之间加一个按钮，显示"📅 单日"或"📅 2日"。点它可以改天数。

**候选池：** 多日模式下，搜索选店不区分第几天，所有选中的店铺进一个"候选池"。底部显示"已选 X 家 | 🪄 智能排程"。

**排程结果：** 点"智能排程"后，显示天卡片——每天一张卡片，里面有时间线摘要。可以左右滑动切换天。

**编辑：** 在天卡片上可以说"换一家""去掉这个""把这个挪到第X天"。编辑只影响当前天。

### 4.4 上下文怎么保留

- **每天的聊天记录独立保存** — 切到第2天，看到的是第2天的聊天，切回第1天，第1天的聊天还在
- **LLM 调用时传入全局信息** — 告诉 LLM "这是3日游的第2天，第1天去了故宫和全聚德，第3天还没规划"，帮助 LLM 理解上下文
- **店铺移动后两边都重排** — 把第1天的店挪到第2天，两边的时间线都会重新计算

| 哪里 | 改什么 | 难度 |

| `server.py` — session_state | 加 trip_mode/trip_days/days/active_day_index | 中 |

| `index.html` — 天卡片 | 排程结果用天卡片展示 | 中 |
| `index.html` — 天标签 | 切换不同天的视图 | 简单 |

## 六、不做的事情（第一版）

- ❌ 跨城市旅游（先只做同城多日）
- ❌ 拖拽排序（成本高，先用文字指令移动）
- ❌ 自定义每天标签名（先用"第1天""第2天"）
- ❌ 多酒店支持（先做一个酒店全程住）
- ❌ 景点开放时间约束（先做简单版，后续加）
- ❌ 多人协作

## 七、怎么验证做对了

| 测什么 | 预期结果 |
|--------|---------|
| 选了5家店（含酒店），设2天，点智能排程 | 自动分成2天，同区域的店在同一天 |
| 排程结果中把第1天的某店挪到第2天 | 两边重新排，时间合理 |
| 切到第2天，说"换一家咖啡" | 只换第2天的，第1天不变 |
| 切回第1天 | 第1天的聊天记录还在 |
| 单日模式→切多日 | 已有店铺进入候选池，不丢失 |
| 多日→切回单日 | 合并所有店铺到一天，弹确认提示 |
